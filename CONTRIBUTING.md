# Contributing

Thanks for your interest in Solana Infra Doctor — an independent, local-first Rust
CLI for Solana RPC production-readiness diagnostics. Contributions of all sizes are
welcome, from typo fixes to new checks.

New here? Start with
[`docs/contributing/first-contribution.md`](docs/contributing/first-contribution.md)
and the [seed issues](docs/contributing/seed-issues.md).

## Quick setup

```bash
git clone https://github.com/satyakwok/solana-infra-doctor
cd solana-infra-doctor
cargo build
```

MSRV is Rust **1.88**. No Solana/Agave SDK is required — the engine talks to RPC
directly via `reqwest`, WebSocket via `tokio-tungstenite`, and Yellowstone gRPC via
`tonic` + `yellowstone-grpc-proto`.

## The checks every change must pass

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
```

Coverage must stay at or above **95%**. If you have `cargo-llvm-cov` installed:

```bash
cargo llvm-cov --all-features --all-targets --ignore-filename-regex 'main.rs' --fail-under-lines 95
```

CI runs the same gates on every PR.

## Run a real local check

Public endpoints, no credentials needed:

```bash
cargo run -- check --rpc https://api.mainnet-beta.solana.com --verbose
cargo run -- compare --rpc https://api.mainnet-beta.solana.com \
                     --rpc https://solana-rpc.publicnode.com --profile bot
cargo run -- ws --rpc https://api.mainnet-beta.solana.com
```

Exit codes: `0` GOOD · `1` WARNING · `2` BAD · `3` UNKNOWN.

## How to add a new RPC check

Single-endpoint checks live in `src/checks/`. In short: add the probe (route it
through `RpcClient` so retries/redaction apply), push an `RpcCheck` with the right
`CheckCategory` and `critical` flag, render it, and test it against the mock HTTP
server in `tests/`. Full walkthrough:
[`docs/contributing/add-a-check.md`](docs/contributing/add-a-check.md).

## How to add or update a workload profile

Scoring lives in `src/compare/scoring.rs`; the profile enum is in `src/cli.rs` /
`src/compare/`. A profile is a deterministic weighting over the same real
per-endpoint data — keep it deterministic and add a `tests/coverage.rs` case that
locks the new ranking behavior.

## How to add example terminal output

Real, public-endpoint captures only. Text examples go in `examples/terminal/`;
screenshots are regenerated via the pipeline in
[`docs/terminal-screenshots.md`](docs/terminal-screenshots.md). See
[`docs/contributing/add-a-report-example.md`](docs/contributing/add-a-report-example.md).

## How to add a redaction-safe real-run report

Markdown reports (`--report`) go in `examples/reports/`. They must be generated by
the real CLI against a **public** endpoint (or a private one with the URL/token
fully redacted), carry a point-in-time caveat, and contain no secrets. See
[`docs/contributing/real-run-evidence.md`](docs/contributing/real-run-evidence.md).

## Architecture

`solana-infra-doctor` is one published crate that is both a library and a binary:

- `src/lib.rs` — the reusable diagnostic **engine**: `checks/`, `compare/` (with
  `scoring`/`render`), `ws/` (with `analysis`), `grpc/`, `rpc/` (with `models`),
  `redact`, `report`, `color`, and shared models. Keep business logic here.
- `src/main.rs` — the thin `sol-doctor` **CLI frontend**: argument parsing,
  dispatch, exit codes only.

Tests must not make live-network calls — use the mock HTTP/WebSocket/gRPC servers in
`tests/`. Determinism is required: verdicts and scores must be reproducible.

## Pull requests

- Keep changes small and reviewable; one logical change per PR.
- Include tests for behavior changes.
- Never log or commit RPC URLs, credentials, query strings, or tokens — redaction
  (`src/redact.rs`) must hold everywhere a URL can appear.
- Keep user-facing errors clear and actionable.
- Update the README / docs and add a `CHANGELOG.md` entry (`## X.Y.Z - Unreleased`)
  for user-facing changes.
- Branch + PR only; do not push to `main`.

## What NOT to contribute (yet)

Out of scope by design — please don't open PRs for these:

- Hosted dashboard, SaaS, hosted monitoring/alerting, account systems, databases.
- Token, NFT, airdrop, points, governance, marketplace, or any speculative mechanic.
- Provider SLA / uptime "guarantees" — outputs are point-in-time diagnostics, not
  guarantees of provider behavior.
- Heavy dependencies (a full Solana/Agave SDK) without a clearly documented need.
- Sentrix or any unrelated product branding; this project is independent and not
  affiliated with Solana Foundation.

## Releasing

Maintainer-only. Releases publish to crates.io via Trusted Publishing from a
manually triggered workflow (no permanent API token in Secrets). See
[`docs/releasing.md`](docs/releasing.md).
